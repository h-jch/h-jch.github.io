背景#

上周的一天线上有个接口突然十几个请求同时全部报错，经排查线上日志发现是出现了数据库死锁

出现死锁后第一时间看了这了接口的代码，这个接口是进行订单相关的校验行为，会查询三张数据表，做四次查询操作，每次查询都使用了select for update加排他锁（这也难怪这个接口这么容易出现死锁了…），并且注意到并不是所有的查询条件都是数据表的索引（这是最大的疑点）

从以上的死锁日志中可以看到

事务1正在等待的SQL语句是：SELECT col1, col2, col3 FROM tableA WHERE col2 = ‘A’ ORDER BY col1 ASC LIMIT 1 FOR UPDATE

持有的锁是行级锁（Record lock），锁在主键索引（PRIMARY），是排他锁，非间隙锁，锁住了col1（主键）=1的这一行记录

正在等待的是col1=3的这一行记录

事务2正在等待的SQL语句是：SELECT (1) AS a FROM tableA WHERE col2 = ‘A’ LIMIT 1 FOR UPDATE

持有的锁是行级锁（Record lock），锁在主键索引（PRIMARY），是排他锁，非间隙锁，锁住了col1（主键）=3的这一行记录

正在等待的是col1=1的这一行记录

而最终的结果就是事务1回滚了

SQL分析#

经过上述的死锁日志分析，我们最终定位到了具体的两句SQL已经发生死锁的记录

但问题是，SQL语句中的查询条件是col2=A，为什么会把col2=B的行记录加上排他锁？

经过一番查询，终于得到问题的答案

生产环境的数据库隔离级别是RC，查询条件col2字段没有索引，在这两个条件下，查询时使用SELECT FOR UPDATE时，会走聚簇索引（主键）的全扫描，会对所有记录加上排他锁后返回给MySQL Server进行过滤，在过滤过程中不满足查询条件的记录的锁会被释放，最终锁住满足条件的行记录

可以理解为有一个对所有记录加锁的过程，如果在这个过程中有任何一行记录已经被其他事务锁定了，那么当前事务的查询会被阻塞，直到这个锁被释放

代码分析#

在发生死锁的接口代码中，这两句查询SQL一前一后执行，最终都会落到数据表中的同一个行记录上，在同一个事务中完全没问题

但是，当两个请求同时进来时，产生两个事务，如果两个事务出现以下的顺序，则会发生死锁

顺序事务1事务2
1SELECT col1, col2, col3 FROM tableA WHERE col2 = ‘A’ ORDER BY col1 ASC LIMIT 1 FOR UPDATE
2SELECT col1, col2, col3 FROM tableA WHERE col2 = ‘A’ ORDER BY col1 ASC LIMIT 1 FOR UPDATE
3SELECT (1) AS a FROM tableA WHERE col2 = ‘A’ LIMIT 1 FOR UPDATE
4SELECT (1) AS a FROM tableA WHERE col2 = ‘A’ LIMIT 1 FOR UPDATE
  1. 事务2首先执行 SELECT col1, col2, col3 FROM tableA WHERE col2 = ‘A’ ORDER BY col1 ASC LIMIT 1 FOR UPDATE; 查询到col2=A的行数据，并对这一行加上排他锁
  2. 紧接着，事务1也执行同样的查询语句，查询过程会扫描全记录，并对记录加锁，所以会对col2=B的行记录加上排他锁。当扫描到col2=A的记录时，发现这一行已经被事务2锁住了，此时事务1会阻塞住
  3. 事务2接着执行 SELECT (1) AS a FROM tableA WHERE col2 = ‘A’ LIMIT 1 FOR UPDATE; 同样扫描全记录，并对记录加锁。此时事务2已经持有col2=A的行记录锁，在扫描过程中需要对col2=B的记录加锁，但是col2=B的行锁已经被事务1持有了
  4. 最终，事务2持有col2=A的锁，需要col2=B的锁，事务1持有col2=B的锁，需要col2=A的锁，形成了死锁局面，最终事务1回滚
